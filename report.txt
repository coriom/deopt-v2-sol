1. protocol code gaps

Item 1
title: Perp matching ingress lacks an emergency stop / guardian layer
severity: critical
why it matters: `src/matching/PerpMatchingEngine.sol` has executor controls but no `guardian`, no pause state, and no emergency kill switch. In production, this leaves the perp order-ingress path without the same fast containment mechanism already present in `src/matching/MatchingEngine.sol`. If an executor key, signer workflow, or upstream matching process is compromised, the perp matching entrypoint itself cannot be frozen independently.
affected files/modules: src/matching/PerpMatchingEngine.sol; operational dependency with src/perp/PerpEngine.sol
blocker before production launch: yes
recommended implementation order: 1

Item 2
title: Collateral universe restriction mode is still missing
severity: important
why it matters: `PROTOCOL_GAP_REVIEW.md` explicitly keeps “collateral universe restriction mode” open, and the codebase currently relies on per-token support plus deposit caps in `CollateralVault` rather than a launch-phase mode that can hard-restrict collateral usage to a narrow approved subset. For a first launch, that leaves too much configuration freedom and increases the chance of activating a token path earlier than intended.
affected files/modules: src/collateral/CollateralVaultStorage.sol; src/collateral/CollateralVaultAdmin.sol; src/risk/RiskModuleViews.sol; src/perp/PerpRiskModule.sol
blocker before production launch: yes
recommended implementation order: 2

Item 3
title: Progressive activation controls and tighter launch-only safety knobs are not implemented
severity: important
why it matters: the repo now has open-interest caps and close-only flags, but there is no explicit staged-activation framework for moving markets/series through bootstrap states. `PROTOCOL_GAP_REVIEW.md` still lists “progressive activation controls” and “tighter launch-only safety knobs” as open. Production launch would otherwise depend on manual operator discipline instead of enforceable code-level activation sequencing.
affected files/modules: src/perp/PerpMarketRegistry.sol; src/perp/PerpEngineAdmin.sol; src/margin/MarginEngineAdmin.sol; src/OptionProductRegistry.sol
blocker before production launch: yes
recommended implementation order: 3

Item 4
title: Underlying-level emergency controls and finer settlement-freeze granularity remain absent
severity: important
why it matters: the code now supports per-market perp close-only and per-series options close-only, but not the “per-underlying emergency controls” and “finer close-only / settlement-freeze paths” still listed in `PROTOCOL_GAP_REVIEW.md`. That leaves incident response coarser than desired: operators can isolate a perp market or an option series, but not cleanly isolate all affected instruments under one underlying or separate more specific settlement freeze scopes.
affected files/modules: src/OptionProductRegistry.sol; src/margin/MarginEngineAdmin.sol; src/perp/PerpMarketRegistry.sol; src/perp/PerpEngineAdmin.sol
blocker before production launch: no
recommended implementation order: 4
